What is it?
Maintenance Agent

A full-stack Swiss property management platform connecting four parties — property managers, building owners, tenants, and contractors — in a single, law-aware system.

Property management in Switzerland involves a dense web of interactions: a tenant files a water damage complaint, which may trigger a rent reduction claim under CO 259a, require owner approval before a contractor is dispatched, generate a QR-bill invoice in the ISO 20022 format, and eventually get posted to a general ledger. Today most managers stitch this together across email, spreadsheets, WhatsApp, and separate billing tools.

Maintenance Agent replaces all of that. It is opinionated about Swiss law, banking formats, and depreciation standards — not a generic platform localised as an afterthought.

🏢
For professional managers
Full operational control: maintenance hub, legal engine, invoicing, leases, asset registry, general ledger — from a single dashboard.
🏠
For private landlords (Phase 2)
Same backend, simplified UX. A private owner managing 1–20 units uses the same platform with a condensed interface and self-service onboarding.

Architecture overview

A monorepo with a strict layered backend, a Next.js frontend, and PostgreSQL via Docker. The backend uses a raw HTTP server — no Express or Fastify — with a custom routing layer.

System topology

LayerTechPort
FrontendNext.js 14, Pages Router3000
Backend APINode.js + TypeScript, raw http.createServer3001
DatabasePostgreSQL 16 via Docker, Prisma ORM5432
Roadmap serverNode.js REST API (scripts/)8111

Backend layers (strict top-to-bottom)

routes  →  workflows  →  services  →  repositories  →  Prisma  →  PostgreSQL
  • Routes — thin HTTP handlers. Parse input, call one workflow, return response. No business logic.
  • Workflows — orchestration. Delegate to services, emit domain events, own status transitions. 28 workflow modules.
  • Services — domain logic. No raw Prisma calls. All side effects go through repositories.
  • Repositories — canonical Prisma access only. Each exports a typed include constant (e.g. JOB_INCLUDE). 27 repository modules.
  • transitions.ts — all 8 status transition maps live here. Nowhere else.

Frontend proxy layer

The Next.js app never calls the backend directly from the browser. Every API call goes through a proxy route in apps/web/pages/api/ which forwards to the backend via the shared proxyToBackend() helper. 197 proxy routes, all conforming. This pattern keeps auth headers server-side and makes the frontend URL surface independent of the backend URL structure.

Why this architecture?

The backend uses a raw http.createServer() rather than Express or Fastify. This was a deliberate early decision: no framework dependency, full control over binary forwarding (PDFs, QR-bill PNGs), and a clean separation between the HTTP layer and the workflow layer. The tradeoff is a bespoke routing and middleware stack — acceptable at the current scale (~225 URL paths), but worth re-evaluating before the team grows significantly. The explicit re-evaluation trigger is documented in the backlog.

The Next.js frontend proxies every API call through its own pages/api/ routes rather than calling the backend directly from the browser. This keeps auth tokens server-side, decouples the frontend URL structure from the backend, and makes it straightforward to add per-route logic (rate limiting, caching) later without changing the frontend components.

Domain event bus

Workflows emit typed domain events (e.g. job.completed, invoice.approved) into the event bus in src/events/. Listeners handle side effects like notification creation and ledger posting. This decouples workflows from cross-cutting concerns.

Auth system

JWT-based. Four token types: authToken (MANAGER), ownerToken (OWNER), vendorToken (VENDOR), tenant session token. Production boot guard refuses to start if AUTH_OPTIONAL=true, DEV_IDENTITY_ENABLED=true, or AUTH_SECRET is missing. Key helpers in src/authz.ts:

HelperUse case
requireAuth()Any authenticated route — returns user or 401
maybeRequireManager()MANAGER or OWNER reads only — never for mutations
requireRole(role)Single role enforcement
requireAnyRole(roles[])Multi-role e.g. ['CONTRACTOR','MANAGER']
requireTenantSession()Tenant portal — returns tenantId or null
getOrgIdForRequest()Returns string | null — null in prod if unauthenticated
Technical
Tech stack

Chosen for compatibility, ecosystem breadth, and a clear deployment path — not because each technology "won" a formal evaluation.

Why this stack — the honest account

The project was built iteratively with AI coding agents (Claude, Copilot). That single fact shaped the stack more than any other consideration: agents produce significantly better output for widely-adopted, well-documented technologies. A niche or emerging stack would have slowed every implementation cycle, since agent training data thins out quickly outside the mainstream.

The second driver was deployment compatibility. The north-star infrastructure is Supabase (managed PostgreSQL), Vercel (Next.js frontend), and Render (Node.js backend). Every technology choice was filtered through that lens — if it didn't have a clear, well-documented path onto that stack, it was deprioritised regardless of technical merit. This is why PostgreSQL was the only database seriously considered (Supabase native), why Next.js was chosen over other React frameworks (Vercel native), and why the backend stays in Node.js/TypeScript rather than exploring Go or Rust.

Tailwind v4 was adopted early — ahead of its stable release — because Next.js and the Vercel build pipeline support it natively, and because agent output quality for Tailwind utility classes is high. The tradeoff is that v4's @theme {} API differs from v3, which occasionally trips up older documentation. Prisma was chosen over alternatives like Drizzle or TypeORM for the same reasoning: it is by far the most represented ORM in agent training data, and its migration tooling (versioned SQL files, shadow DB replay) maps cleanly onto the Supabase deployment model.

In short: the stack was not the result of a benchmarking exercise. It was a deliberate convergence on the set of technologies that would let a small team — heavily augmented by AI — move fast, deploy reliably, and keep operational complexity low.

Frontend design system

Three-layer CSS architecture backed by Tailwind v4.1. All styles trace back to semantic tokens — one token change reskins the entire app.

Three layers

Layer 1 — Semantic tokens
29 CSS custom properties in globals.css @theme {}. Brand (indigo), destructive (red), success (green), muted (slate), surface — each with DEFAULT / light / dark / ring / text variants. Change brand colour in one line.
Layer 2 — @apply classes
84 utility-backed CSS classes in globals.css @layer components. Buttons (.button-primary), tables (.inline-table), tabs (.tab-strip), filters, forms, layout primitives. Use these before writing custom Tailwind.
Layer 3 — CVA primitives
~20 React components in components/ui/. Button (10 variants), Badge, Card, DataTable, Input, Select, ErrorBanner, EmptyState, StatusPill, KpiCard, Modal, ActionBar, ResourceShell, DetailGrid, DetailList. Barrel export via components/ui/index.js.

Key rules

F-UI4: cn() for all dynamic className
Use cn() = twMerge(clsx()) for any className with dynamic values. Template-literal interpolation (className={`... ${x}`}) is banned. A pre-commit hook enforces this.
F-UI4a: Status badges via Badge + statusVariants.js
20 mapper functions (requestVariant, jobVariant, invoiceVariant…). Always <Badge variant={requestVariant(status)}>. Never inline color maps or per-file STATUS_COLORS objects.
F-UI9: Mobile table pattern — dual render
Tables must never be the mobile experience. Use a CSS dual-render: sm:hidden card list for mobile + hidden sm:block table for desktop. Never use overflow-x-auto on tables as a mobile strategy. 18+ pages already migrated.
F-UI11: Tab strips via ScrollableTabs
Never use a bare <div className="tab-strip">. Use <ScrollableTabs activeIndex={...}>. When tabs overflow, it automatically shows a "More ▾" button that opens a BottomSheet — the active tab is always promoted to the visible set.
F-UI8: Shared hooks before custom fetch code
useDetailResource(url) replaces useState+useCallback+useEffect fetch boilerplate. useAction() replaces try/finally pending-state wrappers. ResourceShell replaces loading/error/not-found early-return guards.
F-UI7: English only
All user-visible text, seed data, error messages, and tooltips must be in English. No German, French, or other language strings anywhere until the dedicated i18n epic lands.

Why serial execution (maxWorkers: 1)? Integration tests each spin up a real Node.js server on a dedicated port. Running them in parallel caused intermittent port collision failures that were hard to reproduce and harder to debug — a real incident that cost a full session. Serial execution eliminated the problem entirely with no meaningful impact on total test runtime at current suite count.

Test categories

  • Contract tests (contracts.test.ts) — assert DTO shape and nested relations for key endpoints. Must be updated in the same PR as any DTO change.
  • Integration tests — spin up a real server against the test DB. Most suites fall here.
  • Pure-function suites — domainEvents, httpErrors, orgIsolation, routeProtection, triage — no server spawned, always fast.
  • Strategy engine — 56 unit tests covering all archetype scoring, decision ranking, and cashflow overlay logic.

Developer conventions

These guardrails exist because each one maps to a real outage or a full day of debugging. Read them before making any change.

Schema & database

G1/G8: prisma db push is permanently banned
Always use npx prisma migrate dev --name <desc>. db push creates invisible schema drift that breaks queries silently. CI will fail if db push is detected in any script.
G6: Destructive DB commands require explicit approval
docker-compose down -v, prisma migrate reset, docker volume rm maint_agent_pgdata — these destroy data. Never run without the user explicitly asking. The dev DB is the persistent source of truth.
G2: New model fields — update all consumers in one PR
schema.prisma + migration → repository include constant → DTO interface + mapper → validation schema → OpenAPI spec + api-client → contract test. Miss one and the DTO silently drops data.
G3: Prisma includes must match what mappers access
If a DTO mapper accesses job.request.tenant, the query feeding it must have that relation in its include. Prisma returns undefined (not an error) for non-included relations.
G9: Canonical include constants — no ad-hoc include trees
Every repository exports a typed include constant (e.g. JOB_INCLUDE). All DTO mappers use Prisma.JobGetPayload<{ include: typeof JOB_INCLUDE }>. Never define inline include: { ... } in routes or services.

Code quality

G4: No stub services in production paths
Never leave a function that returns fake data in a file that production routes import. If a real implementation exists elsewhere, re-export it. Stubs must be clearly marked and isolated to test files.
G10: Contract tests must be updated with DTO changes
contracts.test.ts guards the DTO shape of GET /requests, /jobs, /invoices, /leases/:id. Any DTO change must update the contract test in the same PR.

Git & commit safety

G12: Commit every deliverable — no session-long uncommitted work
If you've produced >100 lines of working code without a commit, stop and commit. Code only in the working tree is one editor crash away from being lost.
G13: Frontend + backend = one atomic commit
When a feature spans both layers, both must land in the same commit. The Forecast tab was lost in April 2026 because the backend was committed and the UI wasn't. If the UI is still in progress, commit the backend with a feat: backend (UI pending) message.
G14: Session-end verification
Before closing: git status && git stash list && git diff --stat. If anything meaningful is uncommitted, commit it — even as wip: <description>.

Dev auth tokens

The frontend bootstraps dev JWT tokens automatically in _app.js. Three dev users are seeded: manager@local.dev (MANAGER), dev-owner (OWNER), dev-vendor (VENDOR). All tokens expire 2027-03-15. To regenerate:

cd apps/api
node -e "const jwt=require('jsonwebtoken'); \
  console.log(jwt.sign( \
    {userId:'dev-user',orgId:'default-org',email:'dev@local',role:'MANAGER'}, \
    'dev-secret-key-12345', \
    {expiresIn:'365d'} \
  ))"

Known backlog & tech debt

Active backlog items

  • Azure Document Intelligence tier upgrade — current Free F0 tier allows 500 pages/month. Upgrade to S0 (~$1.50/1,000 pages) before production go-live. Resource: maintenance-agent-docintel, West Europe.
  • Lease signing integration — DocuSign/Skribble wiring requires external API credentials + product decision on provider. Model exists; frontend wiring is the remaining work.
  • Notification push delivery — in-app notifications work; no push/email delivery yet.
  • Reports page — define scope before building (product decision required).
  • Multi-org support — org scoping is in place (M1); full multi-org requires adding orgId to Request (7-step migration documented in SCHEMA_REFERENCE.md) and removing DEFAULT_ORG_ID entirely. Requires product decision on timeline.

Known technical debt

  • Pre-existing test interaction failures (TC-11) — ~10 legacy test suites use local startServer copies instead of canonical testHelpers.ts. They pass individually but can fail in full serial mode. All new tests (since March 2026) use shared helpers.
  • ASSET_TYPE_COLORS in depreciation page — uses hardcoded Tailwind color strings, bypassing the token system. Migrate when depreciation page is next touched.
  • Custom HTTP stack — the raw http.createServer backend is appropriate at current scale (~225 URL paths). Evaluate migrating to Express/Fastify before the team grows or route count exceeds ~200.
  • Dev auth token expiry — three dev tokens expire 2027-03-15. See the Dev Setup section for regeneration command.
