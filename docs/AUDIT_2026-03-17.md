## 1. Executive Summary

**Overall Risk Level: MEDIUM-HIGH**

The project has strong foundational guardrails (G1–G11, F1–F8) and completed a significant security hardening sprint (all 20 SA findings closed). However, the rapid RFP epic development has introduced architectural drift, workflow bypasses, type-safety gaps, and significant documentation staleness. The biggest risk category is **workflow bypass / transition safety** — three invoice mutation routes call services directly instead of their existing workflows, skipping transition guards and domain events entirely.

### Top 5 Findings

| # | ID | Severity | Finding |
|---|-----|----------|---------|
| 1 | **A-1** | **High** | Invoice approve/pay/dispute routes bypass existing workflows — no transition guards, no domain events |
| 2 | **A-2** | **High** | `rfps.ts` service uses 3 `as any` casts to work around Prisma type mismatch — `awardedQuoteId` and `status` fields not in the schema/include |
| 3 | **A-3** | **High** | 25+ lease/legal mutation routes use `maybeRequireManager` instead of `requireRole('MANAGER')` — OWNER can perform MANAGER-only writes |
| 4 | **A-4** | **High** | `leases.ts` service has ~50 direct Prisma calls despite `leaseRepository` existing — largest layer violation in the codebase |
| 5 | **D-1** | **Medium** | PROJECT_STATE.md Document Integrity table has 9 of 13 fields stale (some off by 30–40%) — unreliable as a project contract |

**Merge-safe today?** Conditionally yes for the committed RFP slice (tsc passes, 472 tests pass). But the invoice workflow bypass and `as any` casts in rfps.ts represent latent production risks that should be addressed before any further feature work.

## 2. Confirmed Issues

### A-1 · Invoice routes bypass existing workflows (HIGH)
- **Category:** Backend Architecture / Transition Safety
- **Files:** [apps/api/src/routes/invoices.ts](apps/api/src/routes/invoices.ts#L167-L238)
- **Evidence:** Three routes call service functions directly:
  - `POST /invoices/:id/approve` → `approveInvoice()` (service, L173)
  - `POST /invoices/:id/mark-paid` → `markInvoicePaid()` (service, L196)
  - `POST /invoices/:id/dispute` → `disputeInvoice()` (service, L222)
  - Meanwhile, `approveInvoiceWorkflow`, `payInvoiceWorkflow`, `disputeInvoiceWorkflow` exist in `workflows/` and are exported from `index.ts` but **never imported or called by any route**.
- **Why it matters:** Workflow bypass means no `assertInvoiceTransition()` guard, no domain event emission, no consistent audit trail. An invalid state transition (e.g., DRAFT → PAID) could succeed silently. This was flagged in the prior audit (CQ-11) and remains unfixed.
- **Recommended fix:** Replace service calls with workflow calls in all three handlers. ~15 min fix.
- **Guardrails:** G4, W3, W5, W7

### A-2 · RFP service uses `as any` to access non-existent schema fields (HIGH)
- **Category:** Schema / DTO / Type Safety
- **Files:** [apps/api/src/services/rfps.ts](apps/api/src/services/rfps.ts#L164), [L203](apps/api/src/services/rfps.ts#L203), [L366](apps/api/src/services/rfps.ts#L366)
- **Evidence:**
  - `(rfp as any).awardedQuoteId` — `Rfp` model in schema has no `awardedQuoteId` field
  - `(q as any).status` — `RfpQuote` model in schema has no `status` field
  - Both are accessed via `as any` cast, defaulting to `null` / `"SUBMITTED"` when absent
- **Why it matters:** These casts mask a schema gap. If `awardedQuoteId` and quote `status` were added via migration but the Prisma client wasn't regenerated, this would silently return wrong data. If they were never added, these fields are always null/default. Either way, the DTO advertises fields that don't exist in the data layer.
- **Recommended fix:** Either (a) add `awardedQuoteId` to `Rfp` and `status` to `RfpQuote` via migration + regenerate, or (b) remove these fields from the DTO mapper and handle them differently.
- **Guardrails:** G2, G3, G9

### A-3 · Lease/legal mutation routes use `maybeRequireManager` instead of `requireRole` (HIGH)
- **Category:** Auth / Security
- **Files:** [apps/api/src/routes/leases.ts](apps/api/src/routes/leases.ts) (15+ endpoints), [apps/api/src/routes/legal.ts](apps/api/src/routes/legal.ts) (8+ endpoints), [apps/api/src/routes/rentalApplications.ts](apps/api/src/routes/rentalApplications.ts) (2 endpoints)
- **Evidence:** 25+ mutation (POST/PUT/PATCH/DELETE) endpoints use `maybeRequireManager()` which allows both MANAGER and OWNER roles. Per SA-10 resolution and project rules, mutations should use `requireRole('MANAGER')`. The security-hardening-2 slice fixed this in inventory/requests/tenants/notifications/rentEstimation/financials but missed leases, legal, and rental applications.
- **Why it matters:** OWNER can perform lease creation, lease status changes, legal admin CRUD, signature operations — all intended as MANAGER-only. In OWNER_DIRECT mode this may be partially intentional, but it was explicitly flagged as needing `requireRole('MANAGER')` per SA-10.
- **Recommended fix:** Replace `maybeRequireManager` with `requireRole(req, res, 'MANAGER')` on all mutation routes in leases.ts, legal.ts, and rentalApplications.ts. Or make an explicit architectural decision to allow OWNER writes on these routes and document it.
- **Guardrails:** F1, SA-10

### A-4 · `leases.ts` service: ~50 direct Prisma calls bypassing repository (HIGH)
- **Category:** Backend Architecture / Layer Violation
- **Files:** [apps/api/src/services/leases.ts](apps/api/src/services/leases.ts)
- **Evidence:** `leases.ts` is the heaviest direct-Prisma offender in services (~50 `prisma.lease.*` calls). `leaseRepository.ts` exists with `LEASE_FULL_INCLUDE` and `LEASE_SUMMARY_INCLUDE` but is only partially used. The service also has 10+ `as any` casts for JSON fields (chargesItems, usageFlags, serviceSpaces, etc.).
- **Why it matters:** Any include constant change in leaseRepository won't propagate to the ~50 inline queries. JSON field casts bypass Prisma's type checking.
- **Recommended fix:** Migrate lease service queries to use leaseRepository functions with canonical includes. Address JSON field typing via Prisma's `JsonValue` type or explicit interfaces.
- **Guardrails:** G3, G9

### A-5 · `routes/legal.ts` — 19 direct Prisma calls, largest route-layer violation (HIGH)
- **Category:** Backend Architecture / Layer Violation
- **Files:** [apps/api/src/routes/legal.ts](apps/api/src/routes/legal.ts)
- **Evidence:** 19 direct `prisma.*` calls across legal rules CRUD, category mappings, coverage computation, and evaluation listing. This was the original CQ-1 finding and remains the worst route-layer violation. The rfp-manager-view slice partially addressed RFP-related code but legal admin CRUD is untouched.
- **Why it matters:** Business logic in route handlers violates the layered architecture. Changes to legal data access require touching route files instead of repositories/services.
- **Recommended fix:** Extract to `legalAdminService` + `legalRepository` with canonical includes.
- **Guardrails:** G4, G9

### A-6 · `VENDOR` and `INSURANCE` in STAFF_ROLES but not in Prisma Role enum (MEDIUM)
- **Category:** Schema / Auth Mismatch
- **Files:** [apps/api/src/authz.ts](apps/api/src/authz.ts#L145), [apps/api/prisma/schema.prisma](apps/api/prisma/schema.prisma#L715-L716)
- **Evidence:** `STAFF_ROLES` includes `'VENDOR'` and `'INSURANCE'`. These exist in a separate `StaffRole` enum in the schema (line 700-703) but the primary `Role` enum only has `TENANT | CONTRACTOR | MANAGER | OWNER`. `VENDOR` and `INSURANCE` also appear in the Role enum (lines 715-716) after migration 35, so they exist but no user creation flow uses them. `STAFF_ROLES` references them for notification route access, but no actual users with these roles can be created via the auth system.
- **Why it matters:** Dead code in a critical security path. If a VENDOR user somehow gets created, they'd have notification access but nothing else is wired for them.
- **Recommended fix:** Verify VENDOR/INSURANCE are intentionally in Role enum. If so, add a note. If not, remove from STAFF_ROLES until the roles are fully implemented.
- **Guardrails:** F7

### A-7 · `Rfp.awardedQuoteId` is a plain String, not a Prisma relation (MEDIUM)
- **Category:** Schema Integrity
- **Files:** [apps/api/prisma/schema.prisma](apps/api/prisma/schema.prisma) (Rfp model)
- **Evidence:** `awardedQuoteId` on Rfp (if it exists) and `applicationId` on Lease are plain `String?` fields without `@relation`. Every other FK in the schema uses `@relation` for referential integrity.
- **Why it matters:** No FK constraint means orphaned references are possible. Prisma can't enforce or include these relations.
- **Recommended fix:** Add `@relation` to both fields, or verify they're intentionally denormalized and document why.
- **Guardrails:** G2

### A-8 · 3 test port collisions (3208, 3209, 3211) (MEDIUM)
- **Category:** Test Infrastructure
- **Files:** Multiple test files under `apps/api/src/__tests__/`
- **Evidence:** Ports 3208, 3209, and 3211 are each used by two different test files. Currently safe only because `maxWorkers: 1` enforces serial execution.
- **Why it matters:** If `maxWorkers` is ever increased (for performance), these will cause `EADDRINUSE` failures. The collisions were introduced by the recent RFP test additions.
- **Recommended fix:** Reassign to unique ports or switch to dynamic port allocation (`:0`).
- **Guardrails:** G5, TC-5

### A-9 · `startServer` copy-pasted in 22 test files (MEDIUM)
- **Category:** Test Infrastructure
- **Files:** All integration test files in `apps/api/src/__tests__/`
- **Evidence:** Each file has its own inline `startServer` implementation with minor variations. TC-11 flagged this in the prior audit and it remains unfixed.
- **Why it matters:** Any server boot change requires 22 parallel edits. Different implementations may have subtle behavioral differences.
- **Recommended fix:** Extract shared `startServer`/`stopServer` to `testHelpers.ts`.
- **Guardrails:** TC-11

### A-10 · Public rental application endpoints lack rate limiting (MEDIUM)
- **Category:** Security
- **Files:** [apps/api/src/routes/rentalApplications.ts](apps/api/src/routes/rentalApplications.ts)
- **Evidence:** `POST /rental-applications`, `POST /rental-applications/:id/submit`, `POST /rental-applications/:id/attachments` are all public (no auth) with no rate limiting. `POST /triage` was rate-limited (SA-18) but these similar public endpoints were not.
- **Why it matters:** File upload abuse, application spam, and storage exhaustion are possible.
- **Recommended fix:** Add IP-based rate limiter similar to the `/triage` pattern.
- **Guardrails:** SA-18 pattern

### A-11 · `Notification` model may have schema drift from LKDE `db push` (MEDIUM)
- **Category:** Schema Integrity
- **Files:** [apps/api/prisma/schema.prisma](apps/api/prisma/schema.prisma) (Notification model)
- **Evidence:** Terminal history shows code accessing `n.title` and `n.role` on Notification records, but the Notification model in schema.prisma shows `title` and `role` fields exist (added in LKDE push). If these fields were added via `db push` without a migration, the migration history doesn't know about them. Running `prisma migrate dev` could fail or produce drift.
- **Why it matters:** Unrecorded schema changes are the root cause of the original Feb 24-25 outage. If a fresh database is created from migrations alone, these fields won't exist.
- **Recommended fix:** Run `prisma migrate diff` to verify zero drift. If drift exists, create a reconciliation migration.
- **Guardrails:** G1, G8

### A-12 · `pages/api/contractors.js` — non-conforming legacy proxy (LOW)
- **Category:** Frontend Architecture
- **Files:** [apps/web/pages/api/contractors.js](apps/web/pages/api/contractors.js)
- **Evidence:** 140 lines of hand-rolled `fetch` with manual body parsing, custom error handling, manual header forwarding. Meanwhile, `pages/api/contractors/[id].js` already exists using `proxyToBackend()`. Potential route shadowing.
- **Why it matters:** Maintenance burden, inconsistent error handling, potential route conflicts.
- **Recommended fix:** Migrate to `proxyToBackend()` or delete in favor of existing conforming route.
- **Guardrails:** F3

### A-13 · 300+ inline `style={{}}` objects across frontend pages (LOW)
- **Category:** Frontend Architecture / F-UI4 Violation
- **Files:** Most severe: `admin-inventory/units/[id].js` (~90), `finance/charges.js` (~35), `contractor/index.js` (~30)
- **Evidence:** F-UI4 explicitly bans `style={}` with raw values. Over 300 instances remain, mostly in pages that predate the F-UI4 guardrail establishment.
- **Why it matters:** Style drift, inconsistency, harder to maintain themes.
- **Recommended fix:** Migrate highest-traffic pages first. Prioritize pages modified in recent slices.
- **Guardrails:** F-UI4, F8

### A-14 · Duplicated status color maps across 3 finance pages (LOW)
- **Category:** Frontend Architecture / F-UI5 Violation
- **Files:** `manager/finance/payments.js`, `manager/finance/expenses.js`, `manager/finance/invoices.js`
- **Evidence:** Invoice status color map (`DRAFT: '#...'`, etc.) is copy-pasted in 3 files. Also, 200+ hardcoded hex colors across pages.
- **Why it matters:** Color changes require parallel edits. Violates F-UI5 (shared stateful UI extraction).
- **Recommended fix:** Extract to a shared `statusColors.js` constant.
- **Guardrails:** F-UI4, F-UI5

## 3. Likely Issues / Follow-up Required

### L-1 · Potential LKDE schema drift (needs verification)
- **Severity:** Medium
- **Uncertainty:** Cannot confirm without running `prisma migrate diff` against the live database
- **Evidence:** Multiple schema changes were applied via `db push` during the LKDE epic (documented G8 exception). Notification `title`/`role` fields, `RENT_REDUCTION` enum value, and `LegalRuleScope` enum were all added via `db push`.
- **Verification step:** Run `npx prisma migrate diff --from-schema-datasource ./prisma/schema.prisma --to-schema-datamodel ./prisma/schema.prisma --script` and check for non-empty output.

### L-2 · Domain event types vs NotificationEventType enum divergence
- **Severity:** Low-Medium
- **Evidence:** `DomainEventMap` in `events/types.ts` has 30 event types. `NotificationEventType` enum in schema has 27 values. 12 types exist in code but not in schema; 9 exist in schema but not in code. These are architecturally separate systems (domain events vs notification persistence), but the divergence makes it unclear which events actually create persistent notifications.
- **Verification step:** Trace which domain events trigger notification creation and verify the handler covers all intended types.

### L-3 · `schedulingEscalation` workflow doesn't follow WorkflowContext pattern
- **Severity:** Low
- **Evidence:** Takes raw `prisma: PrismaClient` parameter instead of `WorkflowContext`. No typed Input/Output interfaces. This is a background job function, not a user-triggered workflow, but it's co-located in `workflows/`.
- **Verification step:** Confirm it's only called from background job context, not from route handlers.

### L-4 · `leases.ts` route — `POST /leases/:id/mark-signed` bypasses `activateLeaseWorkflow`
- **Severity:** Medium
- **Evidence:** Sets `status: 'SIGNED'` directly without using `assertLeaseTransition()` or emitting a domain event. The `activateLeaseWorkflow` exists for SIGNED→ACTIVE but there's no workflow for READY_TO_SIGN→SIGNED.
- **Verification step:** Confirm whether a `signLeaseWorkflow` should exist.

### L-5 · Barrel export index missing 3 repositories
- **Severity:** Low
- **Evidence:** `repositories/index.ts` does not export `ratingRepository`, `rfpRepository`, or `schedulingRepository`. These repos must be imported directly by path.
- **Verification step:** Check if any consumer imports from the barrel and misses these.
